# FreStyle 製品ビジョン (B2B SaaS / 新卒研修プラットフォーム)

## 1. 解きたい問題

新卒エンジニアの研修は、企業ごとに以下のような分散したツールスタックで運用されている:

| ツール | 何が載っているか | 痛点 |
|---|---|---|
| Box / Google Drive | 過去のスライド、PDF、Excel 課題 | バージョンが乱立、検索しにくい |
| PowerPoint / Keynote | 講義資料 | 動画化されておらず再利用しにくい |
| Notion / Confluence | 手順書・FAQ | 編集権限が広すぎて荒れがち |
| Slack | 質問・回答 | 流れて消える、過去質問の再利用不能 |
| GitHub | サンプルコード | リンクが散らばり「どれが今期の課題？」が分からない |
| 紙資料 / OJT | コーディング規約、社内ルール | 暗黙知のまま属人化 |

**結果**: 新卒は「ツールを探す」「ファイルを探す」「Slack の過去ログを探す」に時間を浪費し、本来の学習（コードを書く・知識を血肉化する）に到達するまでが遅い。メンターも「あの資料どこだっけ」を毎日聞かれる。

## 2. プロダクトの提供価値

> **「新卒が "学ぶこと" に専念できる単一プラットフォーム。メンターが作った教材・問題・解答が 1 ヶ所に集約され、ブラウザ 1 画面で読む・解く・書く・動かすが完結する。」**

### 提供価値ピラミッド

```
                    ┌──────────────────────┐
                    │ AI による個別学習最適化 │  Phase 3+ (最終形)
                    └──────────────────────┘
                  ┌──────────────────────────┐
                  │ コード実行 / 写経モード      │  Phase 2
                  │ 多言語ランタイム             │
                  └──────────────────────────┘
                ┌──────────────────────────────┐
                │ コース・問題集 CRUD + 進捗管理 │  Phase 1 (= 直近の MVP)
                └──────────────────────────────┘
              ┌──────────────────────────────────┐
              │ 会社単位のデータ分離 (B2B SaaS 基盤) │  Phase 0
              └──────────────────────────────────┘
            ┌──────────────────────────────────────┐
            │ AI ビジネスコミュニケーション練習         │  既存 (Phase -1 とする)
            └──────────────────────────────────────┘
```

## 3. ユーザージャーニー（理想形）

### CompanyAdmin（メンター）

1. SuperAdmin（FreStyle 運営）に紹介され、招待メール経由で会社登録
2. 自社の Trainee を招待（メールアドレスリスト一括）
3. **コース「2026 春新卒バックエンド研修」を作成**
4. セクション「Git 入門」を作り、レッスンを並べる:
   - 📖 Markdown 教材「Git の基本コマンド 10 個」
   - ✅ クイズ「`git add` と `git commit` の違い」
   - 💻 コーディング「与えられた状態から `feature/xxx` ブランチを切り、`README.md` に追記してコミットせよ」
5. ダッシュボードで「3 番目のクイズで全員が間違っている」を発見 → 教材を改善

### Trainee（新卒）

1. 招待メールから初回ログイン
2. 「2026 春新卒バックエンド研修」コースに enrolled
3. レッスンを順番に消化、コーディング課題はブラウザで実行・採点
4. 詰まったら同じ画面の「ビジネスコミュニケーション練習（質問の仕方）」を併用
5. 進捗バーが伸びる

## 4. 既存機能の位置づけ（再定義）

| 既存機能 | 新ポジション |
|---|---|
| AI ロールプレイ練習 (シナリオ) | コース内の **Lesson type=communication** として組み込める。CompanyAdmin が「報連相シナリオを必修にする」運用が可能 |
| 5 軸評価 / Score Card / Learning Report | コース全体の進捗ダッシュボードに統合 |
| Weekly Challenge | 「今週必修のレッスン」リマインダーとして再活用 |
| Daily Goal / Friend / Ranking | B2B 文脈にそぐわないため **MVP では非表示**（残すか削除するかは要相談） |
| Notes | レッスン横の自分メモ機能として継続 |

## 5. MVP スコープと非スコープ

### MVP に含める（Phase 0+1）

- ✅ Company（会社）エンティティと全テーブルの `company_id` 付与
- ✅ ロール: super-admin / company-admin / trainee
- ✅ Course / Section / Lesson の 3 階層
- ✅ Lesson type: reading（Markdown）/ quiz（4 択 + 〇×）
- ✅ 進捗トラッキング (UserLessonProgress, UserCourseEnrollment)
- ✅ CompanyAdmin 用 CRUD 画面 + 進捗ダッシュボード
- ✅ Trainee 用コース受講画面
- ✅ 既存の AI コミュニケーション機能を Lesson type=communication として統合

### MVP に**含めない**（Phase 2 以降）

- ❌ コード実行（次フェーズ最大の山。**MVP リリース後に専用設計で着手、最初から多言語対応**: Python / JavaScript / TypeScript / Java / Go）
- ❌ 写経モード（コード実行の上位機能なので Phase 2 と一緒）
- ❌ AI 採点（既存の AI 評価ロジックの応用なので Phase 2）
- ❌ 認定証 PDF 発行
- ❌ SSO（SAML/OIDC）
- ❌ 課金システム（Phase 2 までは「無償β」として運用）

**コード実行を最初から多言語対応**にする方針:
- 1 言語実装 → 後から多言語に拡張は工数が膨らみがち（言語ごとのテストハーネス、UI 切替、リソース制限の調整）
- 最初から「複数言語を扱う前提」のアーキテクチャを敷く方が安全 → **Judge0 self-host on Fargate** または **商用 API（Sphere Engine / Replit）** を有力候補に検討
- Lambda 言語別ランタイム案は、5 言語以上になると個別管理コストが嵩むため、Phase 2 では採用しない

## 5.5 ターゲット顧客（誰にとっての B2B SaaS か）

**主たる顧客は自社開発企業**。SIer の元請け企業など外部委託主体の組織はターゲット外。

| 想定顧客プロファイル | 例 | なぜマッチするか |
|---|---|---|
| 自社プロダクトを開発する Web / SaaS 企業 | 数十〜数百名規模の事業会社 | 内製エンジニアの早期戦力化を本気で投資する文化がある |
| 社内に教育担当エンジニア（メンター）を確保できる組織 | テックリードや専属メンターのいる会社 | プラットフォームを使い倒す「教材作成側」がいる |
| 新卒・中途を継続的に採用している会社 | 毎年複数名採用 | 教材を一度作れば毎年再利用される |

→ FreStyle 自身が **SaaS 提供事業者**として、これらの顧客企業に対しサブスクリプションでサービスを提供。

## 5.6 オンボーディングフロー（申請 → 承認 → 招待）

セルフサインアップは Phase 3 以降。当面は申請承認制で品質と契約管理を担保。

```
顧客企業 (教育担当)              FreStyle 運営 (SuperAdmin)        新卒 (Trainee)
     │                                 │                                │
     │ 1. ランディングページから利用申請        │                                │
     │   (会社名 / 担当者 / 想定利用人数)      │                                │
     │────────────────────────────────►│                                │
     │                                 │ 2. 内容確認 → 承認                  │
     │                                 │ 3. companies テーブルに INSERT    │
     │                                 │ 4. 初代 CompanyAdmin に招待メール   │
     │ ◄───────────────────────────────│                                │
     │ 5. リンク経由で初回ログイン                │                                │
     │ 6. コース・教材の作成               │                                │
     │ 7. Trainee を招待 (メールアドレスリスト)  │                                │
     │────────────────────────────────────────────────────────────────►│
     │                                 │                                │ 8. ログインして学習開始
```

**実装の段階**:
- Phase 0: 申請フォーム + 承認は手動 (SuperAdmin が SQL を叩く運用) で OK
- Phase 1: 申請フォーム → SuperAdmin 用承認画面 → 自動で会社作成 + 招待メール送信
- Phase 3: セルフサインアップ + プラン選択 + Stripe 決済

## 6. ステークホルダーマップ

| ステークホルダー | 関心事 | 機能要求 |
|---|---|---|
| 新卒 (Trainee) | 早く戦力化したい、評価されたい | 進捗が見える、即時フィードバック、ヘルプが近い |
| メンター (CompanyAdmin) | 教える時間を最小化したい、教材を再利用したい | 教材の使い回し、進捗集計、質問の集約 |
| 教育マネージャー | 育成 ROI、離職率 | チーム全体ダッシュボード、効果測定 |
| 経営層 | コスト、成果 | 会社単位レポート、エンタープライズ機能 |
| FreStyle 運営 (SuperAdmin) | MRR、解約率 | 顧客横断分析、利用量、課金 |

## 7. 競合と差別化

| 競合 | 強み | 弱み | FreStyle の差別化 |
|---|---|---|---|
| Udemy Business | コンテンツ量・グローバル | 自社カスタム教材を作りにくい、新卒研修への最適化 | **メンターがその場で社内向け教材を作れる** |
| paiza ラーニング | コーディング環境・問題質 | コミュニケーション研修ゼロ、自社教材× | **AI ロープレ + コーディング + Markdown 教材を 1 つに統合** |
| Notion + 手作り運用 | 自由度高 | 進捗管理・採点・コード実行が無い | **学習プラットフォームとしての標準機能完備** |
| 内製 Wiki + Slack | 0 円 | スケールしない、属人化 | **構造化された教材＋進捗トラッキング** |

## 8. 成功指標（北極星指標）

- **NSM**: Trainee の "Time to First Productive PR"（入社からチームの本番 PR を出せるまでの日数）短縮
- 補助指標:
  - 1 社あたりの作成コース数（メンターが本気で使うか）
  - Trainee 完走率（>= 70%）
  - 1 セッションあたりの滞在時間（フォーカスできているか）
  - メンターからの NPS（紹介意向）

## 9. リスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| **テナント間データ漏洩** | 致命的 | アプリ層フィルタ + DB 層 RLS の二重防御。CI で漏洩テスト自動化 |
| メンターが教材を作らないと空っぽ | 解約 | テンプレ教材（Git 入門・SQL 入門）を FreStyle 公式コースとして配布 |
| コード実行のセキュリティ | 致命的 | Phase 2 で専用設計（Lambda 隔離、リソース制限、ネットワーク隔離） |
| 学習プラットフォームに対する既存運用との衝突 | 採用率低 | 既存 Notion/Slack からの import 機能を Phase 2 で検討 |
| Bedrock コスト | 利益率低 | 採点回数の制限・キャッシュ |
